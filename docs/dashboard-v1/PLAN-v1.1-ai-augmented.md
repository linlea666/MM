**视觉**：4 行等宽，颜色分明：
- 主力状态：吸筹=亮绿 / 派发=亮红 / 诱多诱空=橙 / 衰竭=紫

## 二、AI 的职责边界（3 条红线）

### 🔴 红线 1：AI 不参与任何"价格数字"决策

**禁止**：
- AI 直接输出入场价
- AI 直接输出止损价
- AI 直接输出止盈价
- AI 给任何行为分数（0~100）
- AI 决定星级

**允许**：
- AI 解释"为什么入场选在这个价"
- AI 润色"为什么是 ★★★★☆ 而不是 ★★★★★"

**理由**：一旦 AI 编价格，幻觉就会直接造成亏损。价格必须由规则层通过可复现的算法算出。

---

### 🔴 红线 2：AI 输出必须携带数据引用

**技术实现**：
```ts
type AIResponse = {
  narrative: string;       // 自然语言段落
  evidence: {
    atom: string;          // 如 "smart_money_cost.Ongoing"
    field: string;         // 如 "avg_price"
    value: number | string;
  }[];
};

// 前端渲染前校验
if (response.evidence.length < 3) {
  return "AI 解盘数据不足，暂不显示";
}

// 每段结论都必须能对上至少一条 evidence
for (const sentence of splitSentences(response.narrative)) {
  if (!hasMatchingEvidence(sentence, response.evidence)) {
    logWarning('unreferenced sentence filtered:', sentence);
    continue;  // 不显示这句
  }
}
```

**Prompt 强制**：
```
你是数据裁决助手，不是预测家。
每个结论后必须附 [证据] 数组，引用具体的原子字段。
如果数据不支持某个结论，必须说"数据不足，无法判断"。
禁止：编造主力意图 / 凭直觉给方向 / 写看起来合理但无数据支撑的句子。
```

---

### 🔴 红线 3：规则层必须独立能跑，AI 宕机不影响决策

**架构约束**：
- 规则层 = 纯函数 `(atoms[]) => Decision`
- AI 层 = 装饰器 `(decision) => decision_with_narrative`
- AI 层崩溃 / 超时 / 限流 → 降级显示规则层原始输出
- UI 上有明确标识：是"纯规则"还是"AI 增强"

**降级表现**：
```
AI 正常：  ┌─ 🧠 AI 解盘 [可用]──────────────────┐
          │ 结论 + 自然语言解读 + 证据           │
          └──────────────────────────────────────┘

AI 异常：  ┌─ ⚠️ AI 服务不可用，仅显示规则结论 ──┐
          │ 结论（简洁版）                        │
          │ [重试 AI] 按钮                        │
          └──────────────────────────────────────┘
```

## 三、AI 的 4 个具体用法

### 用法 ① AI 解盘（手动触发，规则不变）

**触发**：用户点 `[🧠 AI 解盘]` 按钮
**调用频率**：~10 次/天
**模型**：GPT-4o-mini 或 Claude Haiku（快 + 便宜）
**单次成本**：约 0.02 元
**输入**：
```json
{
  "moment": "2026-04-24T10:53:00Z",
  "symbol": "BTC",
  "tf": "30m",
  "current_price": 110800,
  "modules": {
    "participation": { "level": "主力真参与", "score": 0.92 },
    "behavior": { "main": "吸筹", "score": 78, "alerts": [...] },
    "phase": { "current": "真突破启动", "score": 84, "from": "底部吸筹", "next_likely": "趋势延续" },
    "levels": { "R1": {...}, "R2": {...}, "S1": {...}, ... },
    "liquidity": { "up": [...], "down": [...] },
    "decision": { "plan_a": {...}, "plan_b": {...}, "plan_c": {...} }
  },
  "recent_events": [ /* 最近 10 条 */ ],
  "recent_klines": [ /* 最近 10 根 */ ]
}
```

**Prompt 模板**：
```
你是交易大屏的解盘助手。基于提供的模块输出生成 400~600 字自然语言解读。

要求：
1. 严格按 4 段式结构：
   - 主力正在干什么？
   - 现在是什么局？
   - 接下来可能扫哪边流动性？
   - 建议应对？
2. 每段结论后在 [证据] 数组中列出支撑数据
3. 禁止使用"我认为 / 可能 / 或许"等主观词
4. 禁止引用未在输入中出现的字段
5. 如果某段的数据不充分，必须写"数据不足，本段跳过"

输出 JSON 格式：
{
  "narrative_sections": [
    { "title": "主力正在干什么", "text": "...", "evidence": [...] },
    ...
  ]
}
```

---

### 用法 ② 情景 A/B/C 润色（自动触发）

**触发**：状态机阶段切换时 OR 模块分数变化超过 20%
**调用频率**：~30 次/天
**模型**：GPT-4o-mini
**单次成本**：约 0.01 元

**输入**：规则层已经生成的 A/B/C 骨架：
```json
{
  "plan_a": {
    "action": "追多",
    "entry": [110800, 111100],
    "stop": 110300,
    "tp": [111800, 112500],
    "stars": 4,
    "position": "标仓"
  },
  "plan_b": {
    "action": "反手做空",
    "entry": [110500, 110800],
    "stop": 110900,
    "tp": [109600, 108900],
    "stars": 2,
    "position": "轻仓"
  },
  "plan_c": { "action": "观望" },
  "context": { "phase": "...", "behavior": "...", "levels": {...} }
}
```

**AI 任务**：只生成每个情景的**前提条件**和**失效条件**两段文字。价格字段 AI 不能碰。

**输出**（强制 JSON）：
```json
{
  "plan_a": {
    "premise": "价格保持在 trailing_vwap.support 之上 + Fair Value 不拐头",
    "invalidation": "跌破 110300 并在 15min 内未收回"
  },
  "plan_b": { ... },
  "plan_c": { ... }
}
```

**前端合并**：把 AI 的文字字段填入规则层的骨架，渲染成 3 个卡片。

---

### 用法 ③ 冲突二审（仅当规则检测到冲突时）

**触发**：规则层发现两个阶段分数都 >= 60
**调用频率**：~5 次/天
**模型**：Claude Sonnet（质量优先，次数少）
**单次成本**：约 0.2 元

**输入**：冲突的两个阶段 + 各自的证据：
```json
{
  "conflict": [
    {
      "phase": "真突破启动",
      "score": 72,
      "evidence": {
        "resonance_count_1h": 7,
        "power_imbalance": 4.8,
        "ob_broken": true
      }
    },
    {
      "phase": "假突破猎杀",
      "score": 68,
      "evidence": {
        "recent_sweep": { "type": "bearish_sweep", "price": 111200 },
        "price_retrace_pct": 0.85
      }
    }
  ]
}
```

**Prompt**：
```
两个阶段同时触发。基于证据给倾向和理由。
不要编新的证据，只能用输入里的数据。
如果证据不足以分辨，输出 "无法判断，建议降级为无序震荡"。
```

**输出**（强制 JSON）：
```json
{
  "dominant": "真突破启动",
  "confidence": 0.7,
  "reason": "共振 7 次且持续 1 小时，而扫损只有单次，且价格未完全回落到 OB 内。",
  "warning": null
}
```

**规则层处理**：
- AI 选 A + 置信度 > 0.7 → 采纳 A，UI 加 `⚠️ AI 二审` 标
- AI 选 A + 置信度 < 0.7 → 显示 A 但 UI 警告"信号分化"
- AI 返回"无法判断" → 强制"无序震荡"

**关键**：UI 必须显式标明这是"AI 二审结果"，不能让用户误以为是规则结论。

---

### 用法 ④ 日终复盘（每日 00:00 UTC 自动）

**触发**：每日定时
**调用频率**：1 次/天
**模型**：Claude Sonnet（深度推理）
**单次成本**：约 0.5 元

**输入**：当日全部决策记录 + 实际价格走势
**输出**：Markdown 格式复盘文件，存到 `logs/daily-review/2026-04-24.md`

**包含内容**：
- 当日信号一共几次 → 对了几次 / 错了几次
- 错误案例：规则缺陷 还是 市场异常 还是 参数设得不合理
- 对规则层的具体改进建议（阈值调整 / 新增条件 / 缺失的指标组合）
- 当日最值得保留的对信号案例

这是**长期优化规则的最大燃料**。

## 五、Prompt 工程 4 条铁律

1. **输入永远是结构化 JSON**，不是 raw 原子数组。预处理成"模块输出"再喂给 AI
2. **输出强制 JSON Schema**，用 OpenAI 的 `response_format: json_schema` 或 Claude 的 `tool_use` 约束
3. **温度 0.1~0.2**，降低随机性；关键决策场景开 seed
4. **每次 call 都要存原始 prompt/response 到日志**，方便事后查幻觉案例

## 六、为什么坚持规则主导 + AI 辅助

### 规则层的不可替代性
- **可复现**：同输入必同输出 → 回测才有意义
- **可调参**：阈值可写在配置文件里，逐步优化
- **可解释**：每个结论能追到具体条件
- **无成本/延迟**：本地计算 <50ms，AI 最快也要 2s
- **离线可跑**：AI 服务挂了或网络断了，大屏还能用

### AI 层的不可替代性
- **自然语言**：规则做不了"写一段解盘报告"
- **上下文整合**：规则的 if-else 写不过复杂组合
- **长文档**：复盘报告几千字 AI 一次生成
- **演进性**：规则不变时 AI 也能在 prompt 层迭代

### 职责分工总结

```
┌──────────────────────────────────────────┐
│  原子层（纯规则）                        │
│  22 个 endpoint → 23 个原子              │
├──────────────────────────────────────────┤
│  模块打分层（纯规则）                    │
│  行为雷达 / 状态机 / 参与门 / 阶梯…      │
├──────────────────────────────────────────┤
│  决策引擎层（纯规则）                    │
│  → A/B/C 情景骨架（含价格）              │
│  → 星级 / 失效条件 / 仓位建议            │
├──────────────────────────────────────────┤
│  💡 AI 解释层（可选增强）                │
│  → 润色前提 / 失效的文字表述             │
│  → 生成解盘报告                          │
│  → 冲突二审（非决策）                    │
│  → 日终复盘                              │
├──────────────────────────────────────────┤
│  UI 层                                   │
│  → 规则结论（必显）+ AI 文字（可选）     │
└──────────────────────────────────────────┘
```

## 八、需要再决策的 3 个问题

1. **LLM 服务商**：
   - 方案 A：OpenAI（4o-mini + 必要时用 4o）
   - 方案 B：Anthropic（Haiku + Sonnet）
   - 方案 C：国内（DeepSeek / 通义 / 月之暗面），成本降 80%
   - 我的倾向：**方案 C（DeepSeek）做解盘/润色 + Anthropic Sonnet 做二审/复盘**（混合，按质量分层）

2. **AI 调用走服务端还是客户端**：
   - 服务端：API Key 不暴露，但需要自建 proxy
   - 客户端：自用场景其实可以直接调，但 Key 会进浏览器
   - 我的倾向：**服务端**，后端做 proxy + 频次限流 + 响应缓存

3. **AI 响应缓存策略**：
   - 相同输入 5 分钟内 → 复用缓存
   - 避免用户连续点解盘按钮 × 10 次
